## 项目协作规则

- 回复用户必须使用中文，并以“浮生，”开头。
- 用户已确认不需要每次先说明操作方案、不需要修改文件前等待同意、不需要展开判断过程；默认直接执行并给结论。
- 没有要求细节时只给结论，避免啰嗦。
- 执行 shell 命令时优先使用 `rtk`；PowerShell 自带文件读取、枚举命令不需要强行套 `rtk`。

## 当前项目边界

- 不保留旧项目参考资料目录；需要查阅时回旧项目目录查看。
- 新版工作台以 `project.json` 为唯一内部模型。
- 默认导出 YAML 皮肤包，同时支持导出 Jsonnet 源码包。
- 首要产品形态是 GitHub Pages 可部署的纯前端网页工作台。
- 核心工作台功能必须满足 `github.io` / GitHub Pages 运行标准：纯静态可部署，不依赖本地 Node 服务、后端接口、绝对本机路径或只在开发服务器中成立的行为。
- 可选外部服务只能用于访问统计等非核心能力；服务不可用时页面必须降级运行，不能影响编辑、预览、导入导出。
- 布局编辑能力应逐步统一为同一套可视化编辑器；改造任一键盘模块后，需要检查 26 键、数字键盘、符号键盘等相关模块是否可复用并做模块级验收。
- 后续必须支持浏览器本地模板缓存，用户可手动保存模板，不必下载文件。
- 新版工作台必须自包含运行，不依赖旧项目中的示例皮肤、HamsterDoc 或蝦米輸入法目录。

## 功能实现默认约束

- 处理新需求时，以当前需求、当前源码和当前已完成功能为主；除 `GitHub Pages` 兼容、`project.json` 核心模型等稳定硬约束外，不要让历史记忆主导当前实现判断。
- 默认把本工具视为“生成手机端实际可用皮肤文件”的工具，而不是普通网页编辑器；实现、预览、导入导出都要优先服务最终皮肤产物。
- 左侧功能模块的职责是驱动右侧预览；凡是左侧可修改且会影响键盘、工具栏、候选区、状态、交互或布局的内容，右侧都应尽量给出对应的具体预览反馈。
- 右侧凡是已经提供预览的内容，左侧都应有对应的可控功能模块，避免出现“右侧看得到但左侧没地方配”的断层。
- 默认追求高自定义和强联动；新增或改造功能时，不能只做单点生效，还要检查它与预览、导出、导入、不同键盘类型、横竖屏、工具栏、自定义键盘等相关模块的一致性。
- 实现新功能时，要先基于项目中已完成的部分理解用户的真实目标与使用方式，再决定功能如何接入，避免只按字面需求做孤立实现。
- 涉及 UI 或交互改动时，要先拆解参考图或现有成熟界面的布局层级、视觉重点、状态样式、间距节奏和交互反馈，再落到实现，避免凭感觉拼装界面。
