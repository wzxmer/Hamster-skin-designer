# GitHub 运行约束

## 目标

本项目后续默认以“可部署到 GitHub 并直接给用户使用”为目标。

除非明确说明是“开发辅助能力”，否则新增功能都必须优先满足静态托管环境可运行。

这里的“GitHub 可运行”，默认按以下环境理解：

- GitHub Pages 或等价静态托管
- 无自定义后端
- 无本地命令执行能力
- 无本地文件系统读写能力
- 无桌面 App 协议保证

## 设计硬约束

### 允许作为核心能力的部分

- 纯前端页面渲染
- 浏览器内状态管理
- LocalStorage / IndexedDB 草稿保存
- 项目 JSON 导入导出
- 分享码导入导出
- 浏览器端生成 `yaml`
- 浏览器端打包下载 `zip` / `cskin`
- 浏览器内 `jsonnet wasm` 编译

### 不能作为用户主路径依赖的部分

- `apps/builder/server.js` 本地服务
- Node 进程参与的在线功能
- 命令行 `jsonnet` 编译
- 临时目录 `.work/*` 作为用户运行前提
- 本机 `tar` / `zip` 命令
- 依赖桌面文件夹结构的功能

### 只能保留为开发辅助的部分

- `apps/builder/*`
- 命令行构建和调试脚本
- 本地模板抽取、比对、校验脚本
- 面向仓库维护者的批处理工具

## 功能分级

### A 类：必须在 GitHub 静态站可用

- 模板加载
- 默认项目初始化
- 可视化编辑
- JSON 编辑兜底
- 预览
- 版本兼容校验
- 预设加载
- 分享码导入导出
- 项目文件导入导出
- 皮肤产物下载

### B 类：可以输出到皮肤配置，但网页本身不保证真实执行

这类字段属于“生成的皮肤配置能力”，不是“网站运行时能力”：

- `openURL`
- `runScript`
- `openScript`
- `switchRimeSchema`
- 某些 `shortcut`
- `hamster3://...` 相关动作

要求：

- 编辑器里可以编辑
- 预览里可以表达语义
- 导出时必须保留
- 但不能把它们设计成 GitHub 页面必须真的执行

### C 类：开发态可用，生产静态站可降级

- 本地 builder 服务
- 本地命令行编译
- 差异比对脚本
- 模板资产同步脚本

要求：

- 可以继续保留
- 但必须与用户主流程解耦

## 对当前项目的直接影响

### 当前可继续保留

- `apps/web/template-runtime.js`
  这是前端静态运行核心，优先级最高
- `project-data.json`
- `package-assets.json`
- 浏览器侧 `jsonnet wasm`
- 浏览器侧 `JSZip` 打包

### 当前不能再作为长期核心链路的部分

- `apps/builder/server.js`
- `apps/builder/template-service.js`

它们可以保留为开发链，但后续架构不能再围绕它们设计。

## 后续重构原则

### 原则 1

优先把所有用户可见能力收敛到 `apps/web/` 静态运行链。

### 原则 2

新增或重构模块时，先问“GitHub 静态站能不能跑”，不能跑的只允许进入开发辅助层。

### 原则 3

如果同一功能有“本地服务版”和“纯前端版”两条路线，优先纯前端版。

### 原则 4

对外发布能力以浏览器端导出结果为准，不以本地 builder 结果为准。

## 当前建议优先级

1. 继续拆 `apps/web/app.js`
2. 巩固浏览器端模板运行链
3. 把导出与校验都收敛到前端 runtime
4. 再逐步弱化 `apps/builder`
