# 工作台使用引导、方案生成与预设布局规范设计

## 背景

当前 `apps/workbench` 已经具备较多可编辑模块，但存在几个核心问题：

- 左侧模块顺序更偏实现来源，不符合真实编辑路径。
- “使用引导”目前只是组合层入口，不是用户进入工作台后的前置方案生成器。
- 右侧预览为了便于开发，把多种键盘布局同时塞入预览列表，和真实用户方案不一致。
- 复杂模块与基础表单模块的 UI 骨架不统一，重复标题、重复说明较多。
- 布局、划动、长按、工具栏等能力仍部分依赖历史键名与示例皮肤命名习惯，不利于后续方案生成、导入回读和预设维护。
- 现有布局大量来自示例皮肤抽取，但后续应整理为工作台自己的预设布局规范，而不是继续把示例皮肤当事实标准。

本设计用于把工作台重构为“先选方案，再细调皮肤”的产品形态，并建立稳定的预设布局与内部键标识体系。

## 目标

- 把“使用引导”升级为前置方案生成器，决定后续模块是否可用。
- 让右侧预览只显示当前方案相关的键盘，而不是显示所有可选布局。
- 调整左侧模块顺序，使其符合“先基础信息，再布局结构，再视觉主题，最后行为与数据”的真实编辑流程。
- 统一中间编辑区骨架，去掉重复标题、重复说明和重复入口。
- 隐藏底层按键代码名，由工具统一生成稳定内部键标识。
- 建立工作台自己的预设布局规范库，允许从示例皮肤提炼，但不再受示例皮肤历史命名和历史结构直接约束。
- 为导入、编辑、导出提供同一套内部模型，使用户不必理解底层键名或示例皮肤命名。

## 非目标

- 本轮不定义新的视觉风格主题，不扩展超出当前工作台范围的全新业务模块。
- 本轮不继续沿用“示例皮肤里叫什么，工作台就跟着叫什么”的策略。
- 本轮不把左侧导航改成用户可见的一级分组菜单。
- 本轮不要求用户在工作台中直接编辑原始 YAML、Jsonnet 或历史 key code。

## 已确认的产品决策

### 1. 左侧模块顺序

左侧仍保持单列连续模块，不显示用户可点击的大分组，只按内部使用顺序重排。

推荐顺序：

1. `使用引导`
2. `皮肤配置`
3. `导出设置`
4. `键盘高度`
5. `按键边距`
6. `自定义键盘`
7. `按键尺寸`
8. `图片素材`
9. `颜色`
10. `字号`
11. `偏移`
12. `前景缩放`
13. `按键动画`
14. `划动设置`
15. `长按设置`
16. `符号数据源`

视觉上只允许弱分段，不显示“基础信息 / 布局与结构 / 视觉主题 / 行为与数据”这类用户可见标题。

### 2. 中间编辑区骨架

中间编辑区统一为一套固定骨架：

- 页面头：模块名、简短说明、工具按钮。
- 概览卡：只在复杂模块里显示当前对象或当前作用域。
- 主编辑区：放高频可操作内容。
- 补充区：放低频说明、限制和兜底入口。

规则：

- 中间页头保留唯一模块标题。
- 模块正文首卡不再重复模块标题。
- 全局预览规则不在每个模块里重复解释。
- 模块内只保留“当前为什么不能编辑”或“当前正在编辑哪个对象”的差异化说明。

### 3. 主入口与联动展示

每类配置都要有唯一主入口，其他位置只做联动展示，不再承担平行编辑职责。

例如：

- `自定义键盘` 是布局主入口。
- `按键尺寸` 是尺寸主入口。
- `按键边距` 是边距主入口。
- `颜色 / 字号 / 偏移 / 前景缩放 / 图片素材 / 按键动画` 只回答视觉主题。
- `划动设置 / 长按设置 / 符号数据源` 只回答行为与数据。

## 使用引导升级方案

### 1. 角色变化

“使用引导”从普通组合层模块升级为工作台准入层和方案初始化器。

首次进入工作台时：

- 默认只开放 `使用引导`。
- 其他模块处于禁用态。
- 右侧预览不显示正式键盘预览，只显示引导提示态。

用户只有在完成偏好选择并生成方案后，才能进入其他模块。

### 2. 引导需要确认的偏好

引导模块需要确认以下用户偏好：

1. 中文键盘布局选择
   - 提供 `9 键 / 14 键 / 17 键 / 18 键 / 26 键` 等候选方案。
2. 英文键盘选择
3. 符号键盘选择
4. Emoji 键盘选择
5. 是否需要上划功能
6. 是否需要下划功能
7. 是否需要默认工具栏
   - 这里仅决定是否默认生成，后续仍可在自定义布局中手动添加。
8. 空格行需要哪些按键，以及这些按键的顺序
   - 空格行按键相对固定，推荐提供可选键池和已选顺序列表，而不是自由文本。

### 3. 引导页交互结构

引导页建议拆为四段：

- 键盘选择
- 输入行为
- 空格行方案
- 结果确认

最终按钮建议使用明确文案，如：

- 首次进入：`生成我的键盘方案`
- 已有方案后：`重新生成方案`

### 4. 重新生成方案

用户后续可以回到 `使用引导` 重新选择偏好并重建方案，但必须显式触发并二次确认。

确认文案需要明确：

- 会重建当前方案结构。
- 会覆盖布局类设置。
- 主题类配置会尽量保留。

## 工作台状态流

工作台需要显式区分三种状态：

### 1. 未初始化

条件：

- `guide.status = pending`

行为：

- 只开放 `使用引导`
- 其他模块全部禁用
- 右侧显示“请先完成使用引导并生成方案”

### 2. 已初始化

条件：

- `guide.status = ready`

行为：

- 根据偏好生成当前方案
- 解锁其他模块
- 右侧预览只显示当前方案支持的键盘
- 各模块只编辑当前方案中存在的键盘与键身份

### 3. 重建方案中

条件：

- 用户在 `使用引导` 中显式点击重新生成

行为：

- 弹确认
- 确认后重建结构性配置
- 尽量保留视觉主题和可映射的细节配置

## `project.json` 数据模型扩展

建议在 `project.json` 中增加明确的引导结构，而不是把引导状态散落在纯 UI state 里。

```json
{
  "guide": {
    "status": "pending | ready",
    "preferences": {
      "chineseLayout": "pinyin14",
      "englishLayout": "qwerty",
      "symbolLayout": "custom-symbolic",
      "emojiLayout": "emoji-default",
      "swipeUpEnabled": true,
      "swipeDownEnabled": false,
      "defaultToolbarEnabled": true,
      "spacebarRow": ["123", "globe", "space", "symbol", "enter"]
    },
    "generatedPlan": {
      "enabledKeyboards": ["keyboard26", "numeric", "symbolic", "emoji"],
      "previewModes": [
        "pinyin14_portrait",
        "pinyin14_landscape",
        "english_portrait",
        "numeric_portrait",
        "symbolic_portrait",
        "emoji_portrait"
      ],
      "toolbarEnabled": true,
      "swipeProfiles": ["swipe_up"],
      "spacebarRowPreset": ["123", "globe", "space", "symbol", "enter"]
    }
  }
}
```

其中：

- `preferences` 记录用户的原始偏好。
- `generatedPlan` 记录生成后的实际方案结果。
- 后续模块优先读取 `generatedPlan`，而不是直接猜测 `preferences`。

## 方案生成范围

使用引导生成的不是几项零散开关，而是一套完整的“当前方案”。

生成内容至少包括：

- 当前项目启用哪些键盘
- 每个键盘使用哪套预设布局
- 是否生成上划/下划数据骨架
- 是否生成默认工具栏布局
- 空格行按键集合与顺序
- 右侧预览允许显示哪些键盘

生成后，工作台进入“围绕当前方案编辑”的模式。

## 方案重建时的覆盖与保留

### 1. 结构性配置

重新生成方案时默认覆盖：

- 中文 / 英文 / 符号 / Emoji 键盘类型
- 各键盘布局骨架
- 空格行按键集合与顺序
- 是否生成默认工具栏
- 默认划动骨架
- 预览列表集合

### 2. 主题与细节配置

重新生成方案时尽量保留：

- 颜色
- 字号
- 偏移
- 前景缩放
- 图片素材
- 动画
- 已存在且仍能映射到稳定键身份的尺寸、划动、长按和显示配置

## 方案驱动的预览系统

右侧预览需要从“全量开发态预览”改成“方案驱动预览”。

### 当前问题

目前为了方便编辑，工作台把各种键盘布局都塞进预览列表。这样虽然便于开发，但不符合真实用户使用路径，也会干扰“当前到底在改哪套方案”的认知。

### 新规则

- 未生成方案前，不显示正式预览。
- 已生成方案后，预览列表仅来源于 `guide.generatedPlan.previewModes`。
- 用户没有选择某类键盘，则该键盘不出现在预览列表中。
- 后续新增的自定义预览项也必须挂在当前方案下，而不是回到全量模式。

### 对模块的影响

- `自定义键盘 / 按键尺寸 / 划动设置 / 长按设置` 都只围绕当前方案内存在的键盘工作。
- 预览不再承担“展示所有未来可能布局”的职责，而是只展示“当前皮肤方案的工作区”。

## 内部键标识系统

### 1. 基本原则

用户侧不再直接接触历史 key code、示例皮肤命名或手写键名。工作台内部统一生成稳定键标识，导入导出都围绕这套标识工作。

用户看到的应该是：

- 位置
- 作用
- 显示内容
- 动作
- 预览效果

而不是底层键名。

### 2. 标识生成规则

建议为不同区域建立固定命名空间，例如：

- 26 键普通字母键：`button1`、`button2`、`button3`...
- 菜单类键：`menu1`、`menu2`...
- 数字键、符号键、工具栏键、面板键也各自使用固定前缀和编号空间。

工具需要保证：

- 命名规则内部统一生成
- 不让用户手填
- 不再要求用户理解哪个按键该叫什么名字

### 3. 身份稳定优先

用户调整布局时，内部键标识优先保持键身份稳定，而不是按位置重新整体编号。

规则：

- 拖动或交换位置时，原键身份保持不变。
- 插入或删除按键时，只给新增按键分配新标识。
- 不因局部顺序变化而重排整片区域编号。

这样可以最大限度保住该键已有的尺寸、划动、长按、显示和行为配置。

### 4. 键身份元数据

建议每个可编辑键都带一份稳定元数据：

- `id`：内部稳定标识
- `zone`：所属区域，如 `keyboard26`、`toolbar`、`panel`
- `role`：普通键、功能键、切换键、菜单键等
- `layoutSlot`：当前布局槽位
- `display`：显示配置
- `behaviorRefs`：关联到尺寸、划动、长按等配置的引用

后续所有编辑模块都围绕同一个键身份工作，而不是各自猜测 key 名。

## 导入与导出策略

### 导出

工作台内部维护统一键身份，导出时再映射成目标产物需要的结构和字段。

原则：

- 用户不需要知道导出时某个键最终叫什么。
- 导出器根据内部键标识与布局槽位生成目标结构。

### 导入

导入时先解析已有皮肤结构，再识别：

- 普通键
- 功能键
- 菜单键
- 工具栏键
- 面板键

然后回填成工作台内部统一键标识。

目标是：

- 导入后工作台仍以统一模型展示，而不是暴露原始文件中的历史命名。
- 用户可以继续用工作台方式编辑，而不需要理解示例皮肤或外部皮肤的底层命名。

## 预设布局规范化

### 1. 现状问题

当前很多布局和键盘内容来自示例皮肤抽取，这可以作为数据来源，但不能继续作为长期的规范来源。

原因：

- 示例皮肤存在历史命名和历史结构包袱。
- 抽取结果更适合“还原原始皮肤”，不适合作为“工作台自己的预设库”。
- 后续方案生成、预设维护、导入回读都要求一套稳定、可解释、可扩展的内部布局规范。

### 2. 新原则

工作台需要维护自己的预设布局库：

- 可以从示例皮肤提炼
- 但最终按工作台规则统一整理
- 不直接把示例皮肤的命名、目录习惯、零散结构原样沿用

### 3. 预设布局库应包含的内容

建议为每类键盘维护明确的预设集合，例如：

- 中文键盘预设
  - `pinyin9`
  - `pinyin14`
  - `pinyin17`
  - `pinyin18`
  - `pinyin26`
- 英文键盘预设
  - `alphabetic26`
  - 其他需要支持的英文方案
- 数字键盘预设
  - `numeric9`
  - `numericIos`
- 符号键盘预设
  - 系统型
  - 自定义型
- Emoji 键盘预设
  - 系统型
  - 自定义型
- 空格行预设
  - 常见组合和默认顺序
- 工具栏预设
  - 默认按钮集合和排序

### 4. 预设布局的职责

预设布局不只是“几行按键数组”，还需要同时定义：

- 适用键盘类型
- 默认槽位结构
- 默认空格行组合
- 默认功能键集合
- 默认工具栏集合
- 对应需要生成的键身份集合
- 对应需要生成的预览模式

### 5. 与示例皮肤的关系

建议保留“示例皮肤来源记录”，但不让它直接变成用户可编辑事实。

可以这样处理：

- 提取示例皮肤中的可用布局信息
- 转译为工作台内部预设格式
- 以工作台自己的命名和结构存储
- 后续新增预设时优先修改内部预设库，而不是继续去补示例皮肤抽取逻辑

## 模块启用与禁用规则

### 未初始化时

- 仅 `使用引导` 可进入
- 其他模块置灰
- 统一提示：`请先完成使用引导并生成方案`

### 已初始化后

- 全模块解锁
- 但模块内部仍按当前方案决定哪些键盘或对象可编辑

例如：

- 没启用 Emoji，则不显示 Emoji 相关预览和编辑对象
- 没启用工具栏，则不默认显示工具栏编辑对象，但允许在自定义布局中后续手动加入

## 对现有模块的具体影响

### 使用引导

- 从组合层入口升级为前置方案生成器
- 记录用户偏好
- 生成并重建方案
- 展示当前方案摘要

### 自定义键盘

- 只编辑当前方案内存在的键盘布局
- 工具栏仅作为当前布局方案的一部分编辑，不再作为到处重复出现的平行入口

### 按键尺寸

- 仅操作当前方案内键身份对应的尺寸数据

### 划动设置 / 长按设置

- 只对当前方案内存在、且支持该能力的键显示可编辑项

### 符号数据源

- 明确为数据源模块，而不是杂项兜底页

## 验收标准

满足以下条件即认为本设计落地成功：

- 首次进入工作台时，用户无法绕过使用引导直接进入其他模块。
- 完成引导后，工作台会生成一套明确方案，并只显示该方案相关预览。
- 左侧模块顺序符合真实编辑路径，且不引入用户可见一级分组菜单。
- 中间区不再重复显示模块标题和重复说明。
- 用户在工作台里不再需要理解或输入底层按键代码名。
- 导入外部皮肤后，工作台仍能用统一内部模型显示和编辑。
- 预设布局被整理为工作台自己的规范库，而不是继续依赖示例皮肤的历史结构。

## 风险与注意事项

- 方案重建会触碰大量结构性配置，必须先建立稳定键身份映射，否则后续细调配置难以保留。
- 预设布局规范化不能只停留在命名层，必须同时重整布局槽位、默认功能键和预览模式。
- 导入回读需要对历史皮肤结构做归一，否则内部键身份无法稳定建立。
- 左侧顺序调整和模块禁用逻辑会改变当前操作路径，需要同步更新开发文档和后续帮助说明。

## 结论

本轮工作台重构的核心不是单纯美化 UI，而是把工作台从“开发态多布局调试器”重构为“先选方案、再细调皮肤”的正式产品形态。

为此需要同时完成五件事：

- 使用引导前置化
- 方案驱动预览
- 左侧模块顺序重排
- 内部键标识统一
- 预设布局规范化为工作台自己的预设库

只有这五件事一起成立，后续的布局编辑、导入回读、工具栏生成、划动长按保留和多方案扩展才会稳定。
