# 工作台成品化键盘预览渲染设计

## 背景

当前 `apps/workbench` 的右侧预览已经具备较完整的键盘类型覆盖、键位计算和联动能力，但整体更偏“工程预览”而不是“成品键盘预览”：

- 预览首先服务调参，不首先服务成品观感。
- 键盘主体、候选区、工具栏和外层舞台之间的视觉层次还不够接近最终安装后的键盘成品。
- 现有渲染输出虽然可用，但不具备外部参考项目那种“可以直接拿来当预览图”的整体感。

用户这次的核心目标只有一条：

- 右侧预览要尽可能达到参考项目那种键盘预览效果。

同时，当前项目已有明确硬约束：

- `project.json` 仍是唯一内部模型。
- 左侧模块驱动右侧预览的联动关系不能破坏。
- 预览、导出、不同键盘类型、横竖屏、工具栏、自定义键盘之间的一致性仍然要保留。
- GitHub Pages 纯前端约束不能被破坏。

## 目标

- 把右侧预览从“编辑器辅助预览”提升为“接近最终键盘成品的预览”。
- 保持现有 `project.json -> preview` 主链不变，不因为视觉升级而重建整套数据模型。
- 为后续“导出预览图”保留稳定的 DOM 渲染基础。
- 先在最关键的 `26 键竖屏` 上打透成品化效果，再扩展到其他键盘类型。

## 非目标

- 本轮不重做左侧模块结构。
- 本轮不重写 `project.json` schema。
- 本轮不重写 YAML / Jsonnet 导出模型。
- 本轮不直接仿制外部项目的整套实现、状态结构或 UI 布局。
- 本轮不为了成品视觉效果而牺牲既有联动和导出一致性。

## 参考结论

参考项目可借鉴的不是它的具体数据模型，而是它的预览生成思路：

- 预览首先被组织成“像最终键盘”的画面，而不是调试视图。
- DOM 结构天然适合后续截图导出。
- 预览的舞台、留白、比例、顶部区域和键盘整体层级被当作成品的一部分，而不是简单容器。

本项目应借鉴这些方向，但继续以现有 `project.json`、预览计算和键盘语义为真源。

## 方案比较

### 方案 A：仅做 CSS 微调

做法：

- 保持现有预览 DOM 基本不变，只调整样式、间距、背景、阴影和容器大小。

优点：

- 成本最低。
- 对现有逻辑侵入最小。

缺点：

- 预览的结构语义仍偏工程视图。
- 很难稳定达到“成品截图感”。

结论：

- 不采用作为主方案。

### 方案 B：保留数据与计算，重做右侧预览渲染壳

做法：

- 保留现有 `project.json`、键盘模式判断、键位计算、颜色/字号/偏移等数据入口。
- 重新组织右侧预览外层舞台、顶部区域、键盘主体容器和成品视觉层次。
- 在必要时把 `preview.js` 中“计算”和“输出 DOM”之间的边界再抽清一点，但不重建整套预览引擎。

优点：

- 最符合当前仓库状态。
- 能显著提升成品观感。
- 能保留既有联动和导出一致性基础。
- 后续可自然延伸到截图导出。

缺点：

- 需要重整理 `apps/workbench/src/ui/preview.js` 的渲染层级。
- 某些视觉目标可能要求顺手抽出更稳定的中间渲染结构。

结论：

- 采用本方案。

### 方案 C：直接重写一套外部风格预览系统

做法：

- 新建独立预览系统，重新定义 DOM、状态和截图链路。

优点：

- 理论上自由度最高。

缺点：

- 风险最高。
- 极易破坏当前预览与导出、联动之间已经建立的语义一致性。
- 会重复建设当前仓库里已完成的大量能力。

结论：

- 不采用。

## 确认后的总体方案

采用 `方案 B`，并按下面的边界执行：

- 保留 `project.json` 作为唯一内部模型。
- 保留左侧模块和右侧预览的联动主链。
- 保留当前预览模式、键盘类型、候选区、工具栏、面板等基础能力入口。
- 重做右侧“成品化预览”的视觉壳、容器层级和输出结构。
- 如有必要，只抽出预览中间层，不做 schema 级或导出级重构。
- 如果新预览壳稳定，再在其上补“导出预览图”的能力。

## 渲染架构设计

### 1. 保持现有真源

以下内容继续作为真源保留：

- `apps/workbench/src/app.js` 中当前预览状态与模块联动
- `apps/workbench/src/ui/preview.js` 中已有的键盘模式选择、颜色解析、字号解析、布局解析
- `packages/preview-engine/index.js` 中的图标/预览基础能力
- `packages/exporter/index.js` 中与键盘语义直接对应的配置映射

### 2. 新的预览分层

右侧预览改造成三层：

1. `成品舞台层`
   - 负责整体留白、背景、设备感、聚焦感和预览截图氛围。

2. `顶部信息层`
   - 负责候选区、工具栏、展开候选等顶部区域的成品化组织。
   - 这里不再只看“有没有渲染出来”，而是看它在成品里如何呈现。

3. `键盘主体层`
   - 负责按键、排布、功能键层级、明暗模式和整体键盘质感。

### 3. 预览输出目标

`renderPreview(...)` 不再只输出“足够表达参数变化”的结构，而要输出“可以直接作为成品预览图基础”的结构。

这意味着：

- 结构稳定
- 语义清晰
- 样式可控
- 后续可截图

## 分阶段实施设计

### 阶段 1：26 键竖屏成品化

这是本轮第一优先级，也是唯一必须先打透的目标。

范围：

- 重做 `26 键竖屏`
- 重做该模式下的顶部区域和键盘主体成品层次
- 保持现有左改右显、颜色联动、字号联动、偏移联动、按键布局联动

验收：

- 用户第一眼看到的是“最终键盘效果”，不是“参数示意图”
- 与现有数据模型和左侧模块联动不脱节

### 阶段 2：将新壳扩展到同类模式

在阶段 1 稳定后，扩到：

- 英文 26 键
- 9 / 14 / 17 / 18 键中文方案
- 数字键盘

策略：

- 复用同一成品预览壳
- 保留各自原有的键位和布局语义
- 不为单个模式发明独立视觉系统

### 阶段 3：扩展到符号、Emoji、面板和截图能力

在前两阶段稳定后，再处理：

- 符号键盘
- Emoji 键盘
- 面板
- 预览图导出或截图能力

这部分是顺延目标，不作为阶段 1 阻塞条件。

## 代码改造边界

### 主要改动落点

- `apps/workbench/src/ui/preview.js`
- `apps/workbench/styles.css`

### 可能的辅助改动

- `apps/workbench/src/app.js`
  - 仅当预览状态组织需要更稳定的渲染参数时才调整。

- `packages/preview-engine/index.js`
  - 仅当现有图标/基础预览能力不足以支撑成品渲染语义时才补。

### 明确不动的区域

- `packages/project-schema/*`
- `project.sample.json` 数据模型主结构
- 左侧模块顺序与主交互流
- YAML / Jsonnet 导出目标结构

## 风险与应对

### 风险 1：视觉升级带出预览与导出语义漂移

应对：

- 预览重构时同步复核 `packages/exporter/index.js` 中对应键盘语义。
- 不把视觉壳改造误做成键盘语义改造。

### 风险 2：`preview.js` 计算与 DOM 输出耦合过重

应对：

- 允许在 `preview.js` 内做小规模分层整理。
- 不做跨文件大重构，除非已经成为明显阻塞。

### 风险 3：局部模式看起来更好，但系统一致性下降

应对：

- 阶段 1 只先打透 `26 键竖屏`，但设计时提前约束成统一外壳。
- 后续扩展时复用同一视觉语义，而不是每种键盘单独拼装。

### 风险 4：为了效果引入依赖 GitHub Pages 不友好的能力

应对：

- 保持纯前端 DOM/CSS/浏览器内能力。
- 即使后续加截图导出，也只用浏览器侧方案。

## 验证策略

实现阶段至少验证：

- `rtk node --check apps/workbench/src/app.js`
- `rtk node --check apps/workbench/src/ui/preview.js`
- `rtk npm run build`

人工验证重点：

- 左侧高频模块改动是否仍能立刻驱动右侧变化
- 26 键竖屏预览是否明显更接近最终键盘成品
- 明暗模式切换是否仍一致
- 预览放大、候选态、工具栏态是否与新壳兼容

## 成功标准

本轮成功不以“改了多少结构”为准，而以这三个结果为准：

1. 右侧预览明显从“工程预览”提升为“成品预览”
2. 现有 `project.json` 和左侧模块无需重学或重配
3. 后续若要导出预览图，当前 DOM 结构已经可作为稳定基础
